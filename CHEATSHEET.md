# Шпаргалка для сдачи проекта «Логистика 365»

## 1. Что это за проект (30 секунд)

**Логистика 365** — веб-приложение для управления грузоперевозками (TMS — Transportation Management System). Позволяет диспетчерам создавать заказы, назначать транспорт и водителей, строить маршруты, отслеживать выполнение в реальном времени и формировать аналитические отчёты.

**Аналоги:** Wialon, Transporeon, Яндекс.Маршрутизация.

| Уровень | Технология | Зачем |

| **Backend** | Django 4.2 + Python 3.14 | MTV-архитектура, ORM, админка «из коробки» |

| **Фоновые задачи** | Celery + Redis | Уведомления, генерация отчётов |
| **Документация API** | drf-spectacular (OpenAPI/Swagger) | Автодокументирование эндпоинтов |
| **Контейнеризация** | Docker Compose | Воспроизводимое развёртывание |

## 3. Как запустить (пошагово)

```bash
# 1. Перейти в директорию
cd Logistic365

# 2. Активировать окружение
source venv/bin/activate
# или: venv/bin/python manage.py runserver

# 3. Миграции
python manage.py migrate

# 4. Демо-данные
python manage.py seed_data

# 5. Запустить сервер
python manage.py runserver
```

Открыть **http://127.0.0.1:8000/** → логин **admin** / пароль **admin**

> **Если ошибка с venv:** `venv/bin/python manage.py runserver` — запускает напрямую без `source`.

## 4. Архитектура проекта

### 8 модулей (Django apps)

```
logistic365/                    # Главный пакет (настройки, celery, задачи)
├── users/                      # Пользователи и роли (кастомная модель User)
├── core/                       # Дашборд, management-команды
├── orders/                     # Заказы (CRUD, статусы, вложения)
├── transport/                  # ТС + водители + журнал рейсов
├── routes/                     # Маршруты с точками, расчёты
├── tracking/                   # Отслеживание, путевые листы
├── reports/                    # Отчёты, экспорт Excel/CSV
└── references/                 # Справочники (клиенты, грузы, топливо)
```

### Модель данных (ключевые связи)

```
Client 1──N Order N──1 Vehicle N──1 Driver
                │               │
                ├──N Route ──N Waypoint
                │
                └──N TrackingEvent
                └──N Waybill
```

- **Order** — центральный агрегат: клиент, груз, адреса, статус, привязка к ТС/водителю
- **Route** — группировка заказов в рейс с расчётом расстояния/времени/топлива
- **Waypoint** — точки маршрута (погрузка/разгрузка) с координатами и последовательностью

## 5. Ролевая модель

| Роль | Что может |
|------|-----------|
| **Администратор** | Полный доступ, управление пользователями |
| **Диспетчер** | Создавать заказы, назначать ТС/водителей, маршруты |
| **Водитель** | Видит свои заказы, меняет статусы рейса |
| **Бухгалтер** | Просмотр отчётов, экспорт данных (без редактирования) |

Реализовано через поле `role` в кастомной модели `User` (наследование `AbstractUser`).

## 7. Ключевые фичи для упоминания

| Фича | Где реализована |
|------|-----------------|
| **Автогенерация номера заказа** | `Order.save()` — формат `ORD-YYYYMMDD-XXXXXX` |
| **Лог изменения статусов** | `OrderStatusLog` — кто, когда, какой статус |
| **Проверка загрузки ТС** | `Route.validate_load()` — блокирует старт при перегрузе |
| **Расчёт маршрута (Haversine)** | Формула гаверсинусов между координатами точек |
| **Валидация форм** | Серверная (Django) + клиентская (HTML5) |
| **Brute-force защита** | django-axes — 5 попыток, блокировка 1 час |
| **Экспорт Excel/CSV** | openpyxl + csv, кириллица с BOM |
| **Кастомная тема CSS** | Градиенты, анимации, тёмная тема |
| **Context processor** | Счётчик ТС в сайдбаре на каждой странице |

## 8. Типичные вопросы преподавателя и ответы

### Q: Почему Django, а не Flask/FastAPI?
**A:** Django предоставляет «из коробки» ORM, миграции, админку, систему аутентификации, шаблонизатор, защиту от CSRF/XSS. Для CRUD-приложения с ролевой моделью это сокращает разработку в разы. FastAPI подошёл бы для чисто API-сервиса, но здесь нужен и веб-интерфейс.

### Q: Как масштабировать?
**A:** Статика → nginx. Бэкенд → Gunicorn с несколькими воркерами. БД → PostgreSQL с репликацией. Кэш → Redis. Фоновые задачи → Celery с несколькими worker'ами. Docker Compose уже подготовлен.

### Q: Как обеспечена безопасность?
**A:** Хеширование паролей (PBKDF2), защита от CSRF (токены на всех формах), XSS (автоэскейпинг шаблонов), SQL-инъекций (ORM), brute-force (django-axes, 5 попыток), ролевой доступ (проверка `user.role` в views).

### Q: Почему SQLite, а не PostgreSQL?
**A:** SQLite для удобства разработки (не требует внешнего сервиса). В production — переключение на PostgreSQL через одну переменную `DATABASE_URL` в `.env`. `docker-compose.yml` уже содержит PostgreSQL.

### Q: Как работает расчёт маршрута?
**A:** Формула Haversine между координатами последовательных точек. Если координат нет — заглушка 80 км/точка. Для production планируется интеграция с OSRM / Яндекс.Карты API.

### Q: Что такое Celery и зачем он здесь?
**A:** Celery — очередь фоновых задач. Здесь используется для: отправки email-уведомлений, генерации ежедневных отчётов, проверки задержанных заказов, алертов в Telegram. Beat-планировщик запускает задачи по расписанию.

### Q: Как реализована ролевая модель?
**A:** Кастомная модель `User` наследуется от `AbstractUser`, добавлено поле `role` с `TextChoices`. Декораторы `@login_required` и миксины `UserPassesTestMixin` ограничивают доступ. В API — `permission_classes = [IsAuthenticated]`.

### Q: Чем проект отличается от стандартного блога на Django?
**A:** Это не CRUD, а бизнес-приложение со сложной доменной логикой: жизненный цикл заказов (8 статусов), расчёт маршрутов с формулой Haversine, валидация загрузки ТС, генерация путевых листов, экспорт отчётов, интеграция с картами.

## 9. Команды управления

```bash
# Запуск сервера
python manage.py runserver

# Миграции
python manage.py makemigrations
python manage.py migrate

# Демо-данные
python manage.py seed_data --orders 50 --vehicles 15

# Celery worker (нужен Redis)
celery -A logistic365 worker -l info

# Сбор статики
python manage.py collectstatic --noinput

# Тесты
pytest --cov=. --cov-report=html

# Docker
docker compose up --build
```
